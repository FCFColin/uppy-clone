# ADR-014: 多区域拓扑与全局就近路由

## 状态

目标态（非当前部署形态）

> **实现说明**：当前 k8s 多区域 overlay 仅为 stub，并非真正的按区域隔离部署。
> overlay 自身注释亦承认"True per-region isolation is a future goal"。
> 本 ADR 描述的是目标架构，当前单区域部署即可满足学习工程需求。

## 日期

2026-06

## 关联

- ADR-005、ADR-013（部署演进）、ADR-015（CockroachDB）、ADR-016（区域本地房间）

## 上下文

超大型 SaaS 需要全球低延迟、区域故障隔离、数据驻留合规。单区域 GKE + 单主
PostgreSQL 无法满足。

## 决策

1. **多区域 GKE 集群**：每区域一套 StatefulSet（实时层）+ 区域本地 Redis（房间注册表/
   状态缓存，区域内）。
2. **全局接入**：GeoDNS/Anycast + 全局 LB 把客户端导向最近区域；区域内再按
   owner 反向代理路由到房间 owner（ADR-005）。
3. **分布式 SQL**：CockroachDB 多区域集群提供强一致、行级数据驻留（ADR-015）。
4. **跨区控制面**：区域间仅交换控制面信息（房间目录、配置、可观测性聚合），
   通过 mTLS 保护；**不跨区交换实时游戏帧**（ADR-016）。
5. **可观测性聚合**：Thanos/Mimir 或 Prometheus 联邦聚合多区域指标，统一 Grafana。

## 后果

- **优点**：全球低延迟、区域级故障隔离、合规数据驻留、容量随区域线性扩展。
- **代价**：多集群运维、跨区控制面一致性与安全、成本上升。
- **演进**：取代 ADR-013 的单区域 Cloud Run/GKE 形态，控制面/数据面均多活。
