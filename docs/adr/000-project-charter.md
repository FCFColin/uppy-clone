# ADR-000: 项目章程 — 目标与非目标 (Project Charter)

## 状态: 已接受 (2026-06-25)

> 本 ADR 是所有其他 ADR 的**前提**。任何对本项目"复杂度是否匹配问题规模"的评审
> （包括 AI 自动自检），都**必须**先读本章程，再下结论。

## 上下文

本仓库表面上是一个极简的多人网页气球飞行小游戏（单气球物理、15Hz tick、鸟/幽灵 AI）。
但游戏玩法**只是载体**，不是目的。

项目的**真实目的**是：以一个边界清晰、可端到端运行的小业务为载体，**完整实践企业级
多区域、多实例、水平高并发 SaaS 架构**的工程方法论——从领域建模、弹性、可观测、合规，
到多区域部署与运维。

因此，用"一个小游戏需要这么重的架构吗"来衡量本项目是**范畴错误**：游戏规模不是约束，
"把企业级架构练全、练对"才是约束。

## 决策

### 目标 (Goals)

1. **水平扩展**：有状态实时房间在多实例下可水平扩展（实例可寻址 + owner 反向代理 + 租约接管，ADR-005）。
2. **多区域**：就近接入、区域本地房间、跨区域绝不转发游戏帧、强一致持久化（ADR-014/015/016）。
3. **合规**：GDPR 数据删除、审计日志防篡改、数据驻留、威胁建模——作为一等公民存在。
4. **可观测性**：完整 OTel 链路追踪、Prometheus 指标、持续剖析 (Pyroscope)、SLO/告警。
5. **弹性**：熔断、隔板 (bulkhead)、幂等、限流、优雅降级。
6. **解耦与演进**：限界上下文、事务性 Outbox、消息队列 (Redis Stream) 异步化。
7. **工程基线**：高测试覆盖、CI 守门 (lint/audit/索引一致性)、不可变镜像 + 签名、IaC。

### 非目标 (Non-Goals)

- **不**追求游戏玩法深度、关卡或商业化。
- **不**追求最小化依赖或最小化部署成本（与目标 1–7 冲突时，目标优先）。
- **不**以"真实生产流量是否需要"作为裁剪重型组件的理由——本项目按"假定已是超大规模
  SaaS"来设计。
- **不**追求 scale-to-zero / 单机最简部署。

### 刻意保留清单 (Intentional, Do-Not-Flag-As-Over-Engineering)

以下组件是**有意为之的练习目标**，评审/自检时**不得**判定为过度工程：

- 多区域拓扑 + CockroachDB + 全局就近路由（ADR-014/015/016）
- owner 反向代理 + 租约接管（ADR-005，`backend/internal/handler/wsproxy.go`）
- GDPR 硬删除 Worker、审计日志防篡改、威胁模型、数据驻留
- OTel、Pyroscope 持续剖析、完整 SLO/告警/混沌实验文档
- 熔断 / 隔板 / 幂等 / 限流 全套弹性中间件
- 事务性 Outbox + Redis Stream 消息队列
- 多路径认证：QuickPlay + Magic Link + JWT + Refresh + bcrypt Admin + 轻量 RBAC（ADR-026）

## 后果

**正面**
- 设计意图有书面依据，避免每次评审/自检把"刻意的企业级复杂度"误判为过度工程。
- 为"目标架构 vs 已落地现状"提供统一基准（见 [`../architecture/architecture.md`](../architecture/architecture.md) 状态小节）。

**负面 / 需正视的真实债务**
- 章程**不**为真正的技术债背书。以下仍属应清理的债务，与"刻意复杂"无关：
  - 早期平台遗存（Cloudflare `.wrangler/`、孤儿 wrangler 依赖、过时 Cloud Run 措辞）。
  - 文档与现实脱节（把"目标态"写成"已完成终态"）。
  - 仓库卫生（ADR 编号重复、大批改动长期未提交）→ **目录布局**已于 2026-06-27 收敛至 ADR-021（`make check-repo-layout` 门禁）。

## 附录 A：架构决策登记表 (Decision Register)

对每个重型组件给出明确裁决与理由，供未来评审/自检直接引用。

### 保留 (Keep — 刻意的练习目标)

- **owner 反向代理 + 租约接管**（`backend/internal/handler/wsproxy.go`、`internal/game/hub.go`）
  - 裁决：**保留**。
  - 理由：这是"实例可寻址 + 有状态房间水平扩展"的练习核心，也是离开 Cloud Run、收敛
    到 GKE 的根本原因（ADR-005/013）。
  - 与 `/resolve` 的边界：`/resolve` 解决**跨区域**就近定位（返回房间 home region 的
    `ws_endpoint`，ADR-016）；owner 反向代理解决**区域内**连接落到非 owner 实例时的透明
    转发（ADR-005）。二者职责不重叠，并非冗余。
- **多区域 + CockroachDB**（ADR-014/015/016）→ 保留为**目标态**，状态如实标注"提议/部分落地"。
- **合规三件套**（GDPR 硬删除、审计防篡改、威胁模型）→ 保留：明确为学习目标，必须存在。
- **可观测 / 弹性栈**（OTel、Pyroscope、熔断/隔板/幂等/限流）→ 保留。
- **多路径认证**（QuickPlay + Magic Link + JWT + Refresh + bcrypt Admin + 轻量 RBAC，见 ADR-026）
  → 保留。可选未来简化点：若仅作演示可合并为 QuickPlay + Magic Link，但默认不动。

### 清理 (Cleanup — 与目标无关的真实债务)

- Cloudflare 本地遗存 `.wrangler/`、孤儿 `node_modules/wrangler` → 删除。
- 过时 Cloud Run 措辞（代码注释/文档）→ 收敛为 GKE 叙事，ADR-013 保留为历史决策。
- ADR 编号重复（两个 011）→ 去重（bounded-contexts 改为 017）。
- 文档"终态" vs "已落地"措辞 → 统一为"目标架构（部分已落地）"。

## 附录 B：自检指引 (Self-Audit Guidance)

未来对本项目做架构评审或自动自检时：

1. **先读本章程**，确认"刻意保留清单"。
2. 对清单内组件，**不得**以"游戏太小不需要"为由判为过度工程；可评估其**实现质量**
   （正确性、可维护性），但不质疑其**存在性**。
3. 真正应报告的问题限于：附录 A"清理"类债务、实现缺陷、文档与现实脱节、安全/正确性问题。
4. 若认为某个"刻意保留"组件确不合理，应作为**对章程的修订提案**提出，而非直接判为缺陷。
