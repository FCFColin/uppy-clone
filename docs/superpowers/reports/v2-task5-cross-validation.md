### 主题 3: 数据流完整性

| 涉及资产 | 一致性状态 | 发现 | 证据 |
|---------|-----------|------|------|
| A-022 (store) | ⚠️部分一致 | OutboxRepository.InsertOutboxEvent **已包装 circuit breaker**（`r.cb.Execute`），与 v2-R-02 描述"缺 retry.Do 包装"不符（可能 v2-R-02 基于早期版本） | `backend/internal/store/outbox_repository.go:22-32` |
| A-025 (worker) | ✅一致 | GameResultWorker + EmailWorker 走 Redis Stream → XReadGroup → processMessage → XAck，失败 handleRetry 重试 + 死信队列，at-least-once 语义一致 | `backend/internal/worker/game_result_worker.go:123-182`、`retry.go:12-65` |
| A-002 (audit) | ⚠️部分一致 | audit.Log → buffered channel (1024) → processLoop → writeToDB；channel 满同步回退（100ms 超时）；writeToDB 失败仅日志，**不重试，记录丢失**（v2-R-37）；loadLastHash 无超时（v2-R-36） | `backend/internal/audit/audit.go:115-143,164-203` |
| A-015 (outbox) | ✅一致 | Publisher 用 `FOR UPDATE SKIP LOCKED` + 单事务 XAdd + UPDATE processed_at + Commit；崩溃后未 commit 的批次会被重新读取 → at-least-once | `backend/internal/outbox/publisher.go:103-148` |

**主题 3 总结**

- 写路径：handler/game → store.InsertOutboxEvent → outbox.Publisher → Redis Stream → worker → PostgreSQL，**主链路完整** ✅
- at-least-once 语义：
  - Outbox Publisher：`FOR UPDATE SKIP LOCKED` + 事务原子性 → at-least-once ✅
  - GameResultWorker：commit 成功后才 XAck → at-least-once ✅
  - EmailWorker：send 成功后才 XAck → at-least-once ✅
- 失败处理：retry_count 达 maxRetries (5) → 转死信队列（game-result:dead-letter / email:dead-letter）→ XAck 移出主队列 ✅
- 退避策略：`delay = 1<<retryCount * 100ms`（200ms/400ms/800ms/1.6s/3.2s），指数退避 ✅

**新发现（跨层）**：`room_result_async.go` 同时执行三条写入路径：
1. `recordGameResultDirect` → 直接写 PostgreSQL `RecordGameResult`，**失败仅日志，无重试**
2. `enqueueGameResultRedis` → Redis Stream → GameResultWorker（有重试 + 死信）
3. `InsertOutboxEvent` → outbox_events 表 → Publisher → Redis Stream

三写并行设计有缓解措施：GameResultWorker.processMessage 用 `ON CONFLICT DO NOTHING`（基于 `uuid.NewSHA1(gameID+userID)` 生成 result ID），所以 direct write 已写入时 worker 重处理不会冲突。但 direct write 路径本身是冗余的"安全网"，失败后无重试机制，存在数据丢失风险（v2-R-37 audit 类似模式）。

风险点：
- v2-R-41 已确认：at-least-once 语义未在 ADR-009 中文档化
- 新发现：room_result_async.go 三写并行设计未文档化，direct write 与 worker 的去重依赖 uuid.NewSHA1 哈希稳定性，未在 ADR 中说明
- v2-R-02 描述与代码不符：实际已有 circuit breaker 包装，可能 v2-R-02 是发现描述偏差或基于过时代码版本

- **Trusted Proxy CIDR 跨层链路**：
  - 后端 env `TRUSTED_PROXY_CIDRS` ← K8s ConfigMap `trusted-proxy-cidrs` ← overlay `__TRUSTED_PROXY_CIDRS__` 占位符 ← ci-cd.yml:206 `sed` 替换 ← GitHub Secrets `TRUSTED_PROXY_CIDRS`
  - 链路完整但 sed 占位符脆弱（v2-R-18），且 **Terraform 不参与 trusted proxy 配置**（variables.tf 无 trusted_proxy_cidrs 变量），与 K8s 层不一致

- **Redis securityContext 缺失**（v2-R-78）：两个 Redis StatefulSet（redis.yaml + redis-ephemeral.yaml）均无 Pod/Container securityContext，与 balloon-game StatefulSet 安全基线不一致
- **PSS restricted 标签缺失**（v2-R-79）：namespace balloon-game 未声明 `pod-security.kubernetes.io/enforce=restricted` 标签

风险点：
- Terraform 与 K8s trusted proxy 配置不一致：Terraform 不参与，K8s 通过 sed 注入，密钥管理碎片化
- GKE 集群 + VPC 未 IaC 化：灾备切换不可恢复，环境重建需手动操作
- Redis Pod 可能以 root 运行（无 securityContext），与 balloon-game Pod 安全基线不一致，纵深防御有缺口
