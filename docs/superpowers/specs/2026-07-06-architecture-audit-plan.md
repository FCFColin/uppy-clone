# 架构大方向体检计划

## 目标
对项目进行纯审视性的架构大方向体检，覆盖技术栈合理性、模块化程度、演进性、技术债务、架构合规性五个维度。**此阶段不改代码，只产出报告和行动建议。**

## 范围限定
- **第 1 层：宏观大方向体检**（本计划仅执行此层）
- ~~第 2 层：详细技术自检~~（由第 1 层发现的问题驱动，后续另行规划）

## 五个体检维度

### 1. 技术栈合理性评估
检查核心技术选型在当前项目阶段是否仍然合理：
- **Go 1.26 新特性利用** — 搜索 `interface{}`、手动 error 判断、旧 logging，评估泛型/slog/errors.Join 可简化度
- **pgx vs ORM 再判断** — 统计 `store/` 中重复 SQL 模式热力图，估算模板查询 vs ORM 的 ROI
- **Vanilla TS MPA 合理性** — 分析前端 73 文件熵值；评估框架引入（Solid/Svelte）对 Canvas 渲染管线的侵入度
- **Chi middleware 栈复杂度** — 追踪请求经过的中间件链层数；评估 context 传值可追溯性
- **Redis 角色过载** — 统计 Redis 在不同功能域的 key 模式/连接数/数据量级，评估拆分必要性

### 2. 模块化与内聚性评估
检查代码中的物理边界是否匹配设计意图：
- **限界上下文边界** — 跨上下文 import 分析，绘制实际依赖图 vs ADR-017 理论图
- **server 包体重** — 统计 `internal/server` 的文件/函数/注入数，评估拆分 wiring 的必要性
- **前端 game/ 模块化** — 73 文件的分层结构分析，是否可拆为 engine/renderer/network/state 子包
- **共享内核膨胀** — `shared/` 引用方分布分析，检查"工具函数倾倒"倾向

### 3. 演进能力评估
测试架构对变化的适应成本：
- **新增上下文扩散半径** — 模拟"加 Chat 上下文"需修改的文件数和层数
- **基础设施交换代价** — 统计每个基础设施（Redis/PG/OTel）的实现接口数
- **前后端协议同步可靠性** — 检查 constants 代码生成 + CI 校验 + 手动同步的遗漏风险
- **ADR 覆盖度** — 代码中隐含的架构决策 vs 已记录 ADR 对比

### 4. 技术债务全景
量化各项债务的范围和严重程度：
- **死代码** — `make deadcode` + 未引用导出符号 + 废弃 handler/route
- **过度抽象** — interface/struct 比例；分析"仅为测试存在"的接口
- **测试债务** — 各 package 覆盖率热力图；前端 UI 模块实际覆盖估算
- **文档债务** — ADR 逐份验证（决策是否仍执行、前提是否变化、废弃标注）
- **配置债务** — env vars 总数 vs 文档覆盖数 vs 默认值合理性

### 5. 架构合规性评估
验证既定架构规则是否被遵守：
- **Clean Architecture 规则** — 验证 handler/store/auth 的 import 边界
- **环境一致性** — dev/staging/prod 的配置 diff；feature flag 同步状态
- **ADR 采纳率** — 28 份 ADR 逐一标记：✅ 落地 / ⚡ 部分 / ❌ 未执行 / 🔄 废弃

## 产出物
每维度独立报告，合并为《架构大方向体检报告》：
1. 各维度评分（🟢 绿灯 / 🟡 黄灯 / 🔴 红灯）
2. 关键发现与根因分析
3. 建议行动项（按优先级排序，标注预期工作量）

## 执行方式
5 个维度可并行执行，每个维度产出独立子报告。最终汇总。
